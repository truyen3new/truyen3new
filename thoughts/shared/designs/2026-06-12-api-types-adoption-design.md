## Problem Statement

`packages/api-types/` was created as the shared API contract for Light Story but was never wired into any consumer. The codebase has 3 parallel type systems — frontend local types, api-types skeleton, and worker inline types — all drifting independently. Type duplication across `frontend/src/types/`, `frontend/src/lib/apiClient.ts`, and various worker files creates maintenance burden, inconsistent API shapes, and no single source of truth.

## Constraints

- **No big-bang rewrites**. Frontend has 200+ files; we migrate incrementally.
- **OpenAPI spec is the canonical contract**, not hand-written TS. Generated TS is a build artifact.
- **Workers are not primary consumers** of entity/DTO types. They deal with raw HTTP. Include envelope types only.
- **Do not delete `frontend/src/types/`** until the package fully covers everything it exports. Entity types (Story, Chapter, etc.) migrate last.

## Approach

**Phased adoption**: 4 phases, each independently verifiable and deployable.

1. Expand OpenAPI spec to cover all actual endpoints and schemas
2. Restructure package exports so consumers get usable types
3. Wire into frontend build — migrate `ApiResponse<T>` first
4. (Future) Expand to entity types, worker envelope adoption

## Architecture

### Phase 1 — Spec Expansion

Current spec: 5 endpoints, all `type: object` responses, `components.schemas: never`.

Target: Complete spec with:

- **Schemas** under `components/schemas` for every entity and DTO:
  - `Story`, `Chapter`, `Category`, `Author`, `SiteSetting`
  - `ApiResponse<T>` (discriminated: success vs error)
  - `StoryListRequest`, `StoryListResponse`, `StoryDetailResponse`
  - `ChapterListRequest`, `ChapterDetailResponse`
  - `AdminProfileDto`, `AdminAuditLogDto`
  - `AnalyticsDashboardResponse` + sub-types (UserEngagementMetrics, ContentPerformanceMetrics, InfrastructureMetrics, etc.)
  - RPC request types (`LikeStoryRequest`, `IncrementViewsRequest`)
  - `SiteMetricsResponse`, `RoleDistributionResponse`

- **Source**: Lift schema definitions from `frontend/src/types/entities.ts`, `dto.ts`, `analytics.ts`
- **Envelope**: Use oneOf/discriminated union for `ApiResponse<T>` (success shape vs error shape) matching what `dto.ts` already defines via `ApiSuccessResponse | ApiErrorResponse`
- **All endpoint responses** reference `ApiResponse` wrapper schemas
- Remove dead endpoints that don't exist anymore (e.g., old API routes referenced in `endpoint-registry.json`)

### Phase 2 — Package Restructure

```
packages/api-types/
  package.json      ← main: "src/index.ts" (was: "src/generated/index.ts")
  index.ts          ← public re-exports
  openapi.yaml      ← canonical spec
  src/
    index.ts        ← ApiResponse<T>, successResponse(), errorResponse() — keep as-is
    crawler.ts      ← crawler types — keep as-is
    generated/
      index.ts      ← auto-generated by openapi-typescript — do not hand-edit
```

**Package exports**:

```ts
// packages/api-types/src/index.ts (add re-exports)
export { ApiResponse, successResponse, errorResponse } from './index';
export type { Story, Chapter, Category, Author, SiteSetting } from './generated/index';
export type { ApiResponseSchema, StoryListResponse, /* etc */ } from './generated/index';
```

**Design choice**: The package re-exports `components.schemas` types from the generated output. This means:
- Edit `openapi.yaml` → run `npm run generate` → types update automatically
- No hand-written type definitions in the package itself (ApiResponse envelope is the one exception since it's a utility)
- Version bumps in the package propagate type changes across consumers

**Crawler types** (`src/crawler.ts`) stay as-is — they're for external ingestion, not API response types.

### Phase 3 — Frontend Integration

**Build wiring**:

- Add `@light-story/api-types` to `frontend/package.json` dependencies (workspace: `"@light-story/api-types": "*"`)
- In `frontend/tsconfig.json`, add paths: `"@light-story/api-types": ["../packages/api-types/src"]`
- Verify build passes with `npx tsc --noEmit`

**Migration — ApiResponse<T> first**:

1. `frontend/src/lib/apiClient.ts` — remove inline `interface ApiResponse<T>`, import from `@light-story/api-types` instead
2. `frontend/src/types/dto.ts` — import `ApiResponse` from package (keeping current DTOs local)
3. Verify no breaking changes — the package's `ApiResponse` must match the existing shape exactly

**All 200+ frontend files that import from `@/types/dto` or `@/types/entities` continue to work unchanged**. This phase only changes 2 files.

### Phase 4 — Entity Migration (Future)

When the spec has all entity schemas and they're verified to match the actual API:

1. `frontend/src/types/entities.ts` — re-export from `@light-story/api-types` instead of defining locally
2. `frontend/src/types/dto.ts` — import entity types from package throughout
3. Delete the local entity definitions from `frontend/src/types/entities.ts`
4. Workers: optionally add `@light-story/api-types` for envelope types in shared middleware

## Data Flow

```
openapi.yaml (canonical)
    │
    ├── openapi-typescript ──→ src/generated/index.ts (auto-generated)
    │
    └── human review ──→ workers validate against spec
                              │
src/index.ts ──→ @light-story/api-types package
    │
    ├── frontend/src/lib/apiClient.ts ← imports ApiResponse<T>
    ├── frontend/src/types/dto.ts ← imports ApiResponse<T>
    └── (future) frontend/src/types/entities.ts ← re-exports entity types
                              │
                      Next.js build (tsc --noEmit)
```

## Error Handling Strategy

- **Spec mismatch**: If `openapi-typescript` generation fails (invalid YAML, bad refs), the build fails. This is correct behavior — a bad spec should not deploy.
- **Type drift**: The only way types drift is if someone edits `src/generated/index.ts` directly. Prevent this with a CI check: verify the file starts with the auto-generated comment header.
- **Breaking changes**: Since no consumers exist yet, Phase 1-2 are safe. Phase 3 must ensure the package's `ApiResponse<T>` is 1:1 compatible with the existing inline version. The `dto.ts` version uses discriminated union (success vs error); the package version must match.

## Testing Strategy

| Check | How |
|---|---|
| Spec is valid OpenAPI 3.1 | `npx @redocly/cli lint openapi.yaml` (or similar) |
| Generated types compile | `npm --prefix packages/api-types run typecheck` |
| Frontend builds with package | `npm --prefix frontend exec tsc --noEmit` after wiring |
| ApiResponse shape matches | Manual verification — the 2 files in Phase 3 must produce identical types |
| No regression in frontend types | All existing imports from `@/types/*` continue to resolve |

## Open Questions

1. **Should we add a CI lint step** that validates `openapi.yaml` against actual worker routes? Adds complexity but prevents spec drift.
2. **Worker adoption timeline** — Phase 4 or separate? Workers currently don't import the package at all. Adding it would require bundler config changes for esbuild/Wrangler.
3. **Should we drop `endpoint-registry.json`**? It points to Next.js API routes that no longer exist. It's already dead data — we should remove it.
